### DataContextStore

- [ ] Create `DataContextStore` using `data_source_contexts` and `data_source_context_revisions`.
- [ ] Store the current context and ascending retained revision numbers in the versioned `DataSourceContextEnvelope` so transaction callbacks never need unsupported `list()`.
- [ ] Create an empty revision-1 context when a source is created.
- [ ] Write current context and immutable revision in one storage transaction.
- [ ] Retain the latest 50 revisions per source and delete evicted known keys in the same transaction.
- [ ] Implement list revisions and revert-as-new-revision.
- [ ] Delete context/revisions when a source is deleted.
- [ ] Add transaction failure and retention tests.

### DataSourceSecretStore

- [ ] Create the narrow credential wrapper with service `data-source` and versioned account `<sourceId>:password:v<secretVersion>`.
- [ ] Implement get/set/delete/delete-all plus private account-name reconciliation; never expose listing through tools/services.
- [ ] Reconcile and delete unreferenced versions on startup after completing deletion tombstones.
- [ ] Add tests for exact key naming and absence of secrets from returned public views.

### Mutation ordering

- [ ] Repeat candidate test during create/connection-affecting update save, require a reachable result, and bind the persisted test and runtime-created least-privilege acknowledgement to `connectionRevision`.
- [ ] Create source/catalog/initial context/revision atomically after secret write; compensate or reconcile the unreferenced version on failure/crash.
- [ ] Implement password actions `keep` and `replace` with explicit semantics.
- [ ] On replacement, write a new secret version, commit its reference, invalidate the pool, then delete the old version; never overwrite the referenced account first.
- [ ] Persist a disabled `deleting` tombstone before drain/pool/secret cleanup; retain and retry it when secret or metadata deletion fails.
- [ ] Add failure-injection tests for every partial-write boundary.
- [ ] Add crash-restart tests for orphan new versions, old-version cleanup, and deletion tombstone resumption.

## Phase 4: SQL policy and result limiting

### SQL AST policy

- [ ] Create `SqlReadOnlyPolicy` with explicit PostgreSQL and MySQL dialect selection.
- [ ] Import only required `node-sql-parser` dialect code if supported by the bundler.
- [ ] Require exactly one parsed statement and a top-level read query.
- [ ] Recursively reject DML, DDL, data-modifying CTEs, `SELECT INTO`, procedure/execute syntax, copy/load/outfile, session/transaction control, permission changes, and locking reads.
- [ ] Collect normalized referenced objects from the AST.
- [ ] Apply allowed namespace/object checks after dialect-specific identifier normalization.
- [ ] Return a typed validation result suitable for query wrapping.
- [ ] Make validation synchronous so risk assessment can validate before approval; re-run it in the handler/runtime before execution.
- [ ] Serialize the accepted AST to `safeSql` and prove placeholder order is preserved.
- [ ] For PostgreSQL allow reused placeholders but require a contiguous `$1..$n` set; for MySQL require exact `?` occurrence count, using AST/token data rather than raw regex.
- [ ] Fail closed for unsupported valid SQL shapes.

### Parser security corpus

- [ ] Add ordinary select, CTE select, joins, unions, aggregates, windows, subqueries, and parameter tests.
- [ ] Add semicolon/multiple-statement tests with whitespace and comments.
- [ ] Add PostgreSQL data-modifying CTE and `SELECT INTO` tests.
- [ ] Add MySQL outfile/dumpfile/load and locking-read tests.
- [ ] Add quoted, mixed-case, escaped, and schema-qualified allowlist bypass tests.
- [ ] Add function-call tests documenting that functions are allowed but not considered a security boundary.

### Parameter codec

- [ ] Implement typed parameter conversion for string, number, boolean, null, and date.
- [ ] Validate PostgreSQL `$1..$n` placeholders and MySQL `?` count.
- [ ] Reject invalid/NaN/infinite numbers and invalid dates.
- [ ] Never log parameter values.

### Result limiter

- [ ] Create `DataResultLimiter` with row, cell, and 40,000-character serialized limits.
- [ ] Remove the `maxRows+1` sentinel row and set `truncated=true`.
- [ ] Truncate textual cells over 4,000 characters with explicit metadata.
- [ ] Stop adding rows before serialized output can reach WorkX's 50,000-character persistence threshold.
- [ ] Apply deterministic pagination/40,000-character response caps to agent list/describe outputs too.
- [ ] Document/test the MVP limitation that one pathological cell can allocate driver memory before post-decode truncation; do not claim a pre-decode byte bound.
- [ ] Add deterministic tests for multibyte text, JSON, many columns, and one giant cell.
