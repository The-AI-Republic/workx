# Track 14: Plan Review

**Priority: P1** · **Effort: M–L** · **Status: READY TO IMPLEMENT**

> Source: second-pass claudy↔browserx research (2026-05-14), implementation-readiness + multi-platform pass (2026-05-15), deep-implementation pass (2026-05-15: full read of claudy's plan machinery + a verified trace of browserx's *actual* approval surfacing on all three platforms), terminology + decoupling pass (2026-05-15: renamed **Plan Mode → Plan Review** to free the per-session "mode" axis reserved by PR 223/228 for agent operating modes), and **end-to-end all-phases pass** (2026-05-15: every phase closed against verified plumbing — the feature now lives entirely on `ToolRegistry` with **zero `ApprovalGate` changes**, a single sufficient choke point, closure-wired tools, a `PromptComposer` fragment for cross-turn guidance, `/plan` as a `PromptCommand`, the plan as a structured tool input, and a `plan_artifact` rollout record; Phase 3 reuses the existing approval transport; Phase 4 staleness is explicitly advisory). ## Problem

BrowserX gates risky actions **one at a time by numeric risk score** (`ApprovalGate.check()`). There is no "propose a full plan → freeze every mutation → one approval → execute" workflow. For an agent taking real-world web/desktop actions (multi-step automation, form fills, downloads, purchases, file ops), per-action prompting is noisy and low-trust: the user never sees and approves the *whole intended sequence* before anything happens, and cannot edit it. **Plan Review** turns N reactive approvals into one informed, editable approval — but only if (a) the freeze is airtight, (b) it is cleanly reversible, and (c) it cannot strand an agent in an unleavable state. Browserx's three deploy targets make (c) the hard part: the approval surface that lifts the freeze is **not reachable the same way on all three**, and on the headless server it is, today, **not reachable at all** for this code path (see "The real seam" and Decision 3 — the central finding of the deep pass).

> **Naming.** This was "Plan Mode" in the first three passes. PR 223/228 reserve **"mode"** for the per-session *agent operating mode* axis (code mode vs. general automation — the `MODES` registry / `composeMainInstruction` / `ModeChanged` path). To avoid a terminology collision *and* to model the feature correctly, it is renamed **Plan Review** and modelled as a distinct, orthogonal **`ToolRegistry` state** (a `planReviewActive` boolean on the object that owns the tool-execution choke point) — **not** a new `ApprovalMode` value and **not** an `ApprovalGate` change. State vocabulary: the session is *review-pending* while the freeze is active. Tools: **`BeginPlan`** (enter) / **`SubmitPlanForReview`** (request approval). Command: **`/plan`** (retained).

## What Claudy Does

Claudy's plan mode is a **permission mode** with restore semantics. We deliberately **do not** copy that shape (see the Naming note + Decision 2); the transferable *ideas* (verified by full read) are:

1. **A reversible freeze.** Claudy flips `toolPermissionContext.mode='plan'`, stashes `prePlanMode`, and restores it on exit with a circuit-breaker downgrade. **We diverge:** Plan Review is an *orthogonal boolean on `ToolRegistry`* (the tool-execution choke point); no `ApprovalMode`, no `ApprovalGate` change, so there is **nothing to restore** — exit just clears the boolean. This eliminates claudy's entire restore/circuit-breaker/`prePlanMode` surface (and a class of restore bugs) and is also forced by the reserved-"mode" constraint.
2. **An entry tool that is auto-allowed and injects a strong read-only instruction.** `tools/EnterPlanModeTool/EnterPlanModeTool.ts`: `isReadOnly()=>true`, no `checkPermissions`/`validateInput` (auto-allowed), `shouldDefer:true`, and `mapToolResultToToolResultBlockParam` injects the verbatim *"DO NOT write or edit anything yet… present your plan for approval. This is a read-only exploration and planning phase."* as the tool result, reinforced by a recurring attachment.
3. **An exit tool that validates cheaply before any dialog, asks a human, re-persists edits, and echoes the plan back.** `tools/ExitPlanModeTool/ExitPlanModeV2Tool.ts`: `validateInput()` **rejects before the approval dialog** if not in the plan state (cheap guard against the deferred tool being called post-exit); `checkPermissions()` → `{behavior:'ask'}`; a user-edited plan arrives via `permissionResult.updatedInput.plan`, is re-persisted, and the result echoes the **full approved plan** back to the model labelled *"(edited by user)"* when changed (this echo is the load-bearing way the model receives the plan — claudy does **not** rely on a side-channel).
4. **The symmetric trap-guard — never expose the entrance when the exit is non-functional.** Both tools share an identical `isEnabled()` that returns **false when `--channels` is active** (the approval dialog needs the terminal). A disabled exit ⇒ a disabled entrance, enforced at *tool-availability time*. The single most important pattern to port.
5. **Optional scoped semantic grants.** `ExitPlanMode.allowedPrompts` → converted to **session permission `addRules`** at exit and **re-applied across a context clear**, so the planning-phase grant carries into execution without re-prompting.

`utils/planModeV2.ts`'s interview phase, multi-agent Explore/Plan counts, the filesystem `plans/` slug-file artifact, the "edit only the plan file" rule, and the shell/code-exec "dangerous permission" strip/restore are **coding-agent-specific and are NOT ported** (see Decision 2 and Risks).

### Per-Platform Behavior

Plan Review's load-bearing axis is **how the single "Submit plan for review?" approval resolves**. The categorical freeze is identical core logic everywhere it runs; the approval is not.

- **BrowserX (extension).** Core `ApprovalManager` → `ApprovalRequested` event → webfront card → `ExecApproval` op → `handleExecApproval` → `handleDecision`. A human reviews, optionally **edits**, approves/rejects. Balanced mode uses `getTimeoutForMode()===0` ⇒ the request waits indefinitely for the human (no auto-resolve) — correct for an interactive surface. Fully supported. Canonical experience.
- **Apple Pi (desktop).** Structurally identical to extension (same core manager, same `EventProcessor` card, same `ExecApproval` op). One implementation covers ext + desktop.
- **Apple Pi Server (headless).** **No working core-manager round-trip exists today.** `ApprovalGate` is not constructed server-side, and even if it were: in `balanced` the core manager's `requestApproval` returns a *timer-less* promise (`getTimeoutForMode()===0`) that **hangs forever** — a permanent trap, not a timeout; in `high_speed` the 10-minute timer fires and **auto-APPROVES** (`decision:'approve'`, `metadata:{timeout:true}` — `src/core/ApprovalManager.ts:145-184`), i.e. fail-**OPEN**, silently lifting the freeze. There is **no decision return-wiring for the core manager on the server at all** — a connected operator cannot approve it either (only the unrelated exec manager has `resolveApproval` wired, `ServerAgentBootstrap.ts:506-510`). **Conclusion:** on the headless server, human-approved Plan Review is non-functional and high-speed is a security hole. Therefore Plan Review is **registration-gated OFF on `platformId==='server'`** and enabled there only by an explicit Track-20 managed plan-resolution policy that resolves the approval *by policy* (allowlist auto-approve / auto-deny), never by a human round-trip. This is browserx's honest analog of claudy's `--channels` guard, and *stronger* than the first-pass "ApprovalManager solves the trap for free" claim, which was false for the server.

### Key design decisions (and divergences from claudy)

1. **One sufficient choke point: a freeze guard in `ToolRegistry.execute`, keyed off Track 02 — not the risk score, not a mode, not the ApprovalGate.**
   - **The single enforcement point.** Every tool call on every platform — local tools, MCP `browser__*`, sub-agent calls — funnels through `ToolRegistry.execute(request)` (`src/tools/ToolRegistry.ts:316`; verified sole caller path: `TurnManager.ts:1164`, MCP via registered handlers, sub-agents via `buildSubAgentInvoker`). It already has a `preExecuteCheck?: PreExecuteCheck` hook (`:97`, setter `:128`) invoked at `:369-385`, **before** the optional `if (this.approvalGate)` block (`:388`). The freeze is a registry-native check at exactly that position: `if (this.planReviewActive && !this.isReadOnly(request.toolName, request.parameters)) return APPROVAL_DENIED('plan-review-freeze')`. `this.isReadOnly` is Track 02, **native to the registry on every platform incl. server** — so **no injected classifier and no second ApprovalGate branch are needed**. This is a deliberate simplification over the earlier "two co-equal choke points + inject a `readOnlyClassifier` into `ApprovalGate`" design: one enforcement point, one signal, zero new cross-object wiring. **Divergence:** claudy enforces via `toolPermissionContext.mode` inside its permission layer; browserx's correct equivalent is the universal tool-execution funnel, which `ApprovalGate` is not (it isn't even constructed on the server).
   - **Why not the `ApprovalGate.check()` branch the earlier passes proposed?** It is redundant (the registry guard runs first and denies before the gate is consulted), platform-partial (`ApprovalGate` is absent server-side), and would need the cross-object `readOnlyClassifier` injection. Dropped. An optional gate-side branch may be added later purely for approval-surface telemetry, but it is **not** part of the enforcement design and not required for correctness.
   - Read-only browser ops (`browser_dom snapshot`, `page_vision screenshot`, `web_scraping`/`data_extraction`* reads, `browser_navigation getHistory/getCurrentUrl`, `cache_storage_tool read/list`, `setting_tool get/list`) pass; `click`/`type`/`keypress`, `form_automation`, downloads, `navigate`/`reload`, `cache_storage_tool delete`, terminal writes, and purchases (**Track 23**) freeze. (*`data_extraction` is force-frozen until Track 27 — see the Classification Audit.)

2. **Orthogonal state on the `ToolRegistry` (the enforcement owner) + a session-store plan artifact (no `prePlanMode`, no `ApprovalGate` change at all, no filesystem file, no Track-13 context InputItem).**
   - Add `planReviewActive = false` + `beginPlanReview()`/`endPlanReview()` (idempotent set/clear) **to `ToolRegistry`** — the object that already owns the choke point and `isReadOnly`. **`ApprovalGate` is not modified at all**: no `ApprovalMode` member, no `prePlanMode`, no restore-anchor, no circuit-breaker, no injected classifier, no `getAskThreshold`/`getTimeoutForMode` edits. The single source of truth is the registry flag; the Begin/Submit tool closures call `registry.beginPlanReview()/endPlanReview()` directly. (This is the full decoupling payoff: the underlying `ApprovalMode` is never touched, so there is nothing to restore, and the reserved-"mode" collision plus the earlier passes' exhaustiveness churn vanish entirely.)
   - The plan is a **session-scoped artifact persisted as a new `RolloutItem` variant** (`storage/rollout/types.ts:162-169` is a closed union with **no plan type today** — add `plan_artifact` + zod schema + writer/reader; net-new, not a drop-in). **Divergence:** claudy's `~/.claude/plans/<slug>.md` filesystem model does not port — browserx has no project root or durable per-session file area; the rollout store is the correct substrate and gives Track 15 (rewind) a clean record to discard.
   - **Correction vs first-pass:** the model receives the approved plan via the **`SubmitPlanForReview` tool-result echo** (claudy's actual mechanism — full plan text, `"(edited by user)"` when changed), **not** via a `context` InputItem. Track 13's `context` item carries only a `path`, renders as the literal text `[context: <path>]` (`Session.ts:673-677`), and has no content resolver/funnel (Track 13 funnel is DESIGN-ONLY). The echo is self-contained and platform-uniform; the rollout record is for durability/rewind, not model delivery.

3. **Begin/Submit as closure-wired tools (not generic registrar tools — `ToolContext` exposes no gate/manager handle).**
   - **The plumbing constraint (verified).** `ToolContext` is `{sessionId,turnId,toolName,callId,metadata,onProgress}` (`BaseTool.ts:166`) — a tool handler gets **no** reference to the registry, gate, manager, or session. So `BeginPlan`/`SubmitPlanForReview` cannot be plain `BaseTool`s that "reach out". They are registered as **handler closures built in the same place `new ApprovalGate(...)` is** — `service-worker.ts:93-118` (ext) and `DesktopAgentBootstrap.ts:232-257` (desktop) — where `registry`, `approvalGate`, and `approvalManager = agent.getApprovalManager()` are all in scope. The closure captures `registry` + `approvalManager`.
   - **`BeginPlan` closure**: `registry.beginPlanReview()`; return the browser-domain read-only instruction as the `ToolResult` (port claudy's tool-result text: *"…take only READ-ONLY actions — navigation history, DOM snapshots, screenshots, scraping reads. DO NOT click, type, submit, download, purchase, or change settings yet. Use SubmitPlanForReview to present your plan for approval."*). `runtime.concurrency.isReadOnly:()=>true` so `BeginPlan` itself is never frozen.
   - **`SubmitPlanForReview` closure**: cheap reject if `!registry.planReviewActive` (claudy's pre-dialog guard — the deferred tool can be re-announced post-exit; reject *before* surfacing approval). Otherwise persist the `plan_artifact` (status `submitted`), then `await approvalManager.requestApproval({ type:'dangerous_action', title:'Submit plan for review?', details:{action:'plan_review', parameters:{plan}}, metadata:{...}, timeout:0 })` — the public method (`ApprovalManager.ts:101`), the **verified working ext/desktop round-trip** (event → webfront card → `ExecApproval` op → `handleExecApproval` → `handleDecision` resolves the awaited promise). On `approve`: `registry.endPlanReview()`, update the artifact (`approved`/`edited`), **echo the approved plan back** as the `ToolResult` (claudy's load-bearing delivery; `"(edited by user)"` when changed). On `reject`: `endPlanReview()`, artifact `rejected`, echo the rejection + (Track 15) the rewind anchor.
   - **The trap-guard.** No per-tool `isEnabled()` hook exists (`BaseTool`/`ToolRegistry` lack it; only `CommandBase.isEnabled?()`). Claudy's "filter the tool out of the list" is ported as **registration-time gating**: the closures are simply **not wired in the server bootstrap** (`ServerAgentBootstrap`) unless a Track-20 plan-resolution policy is present; ext/desktop always wire them. A defensive refusal inside the `BeginPlan` closure (`platformId==='server'` && no policy ⇒ explanatory error `ToolResult`, do **not** flip `planReviewActive`) means a stale/buggy registration still cannot strand the agent. **Refined divergence (corrects the first two passes):** "ApprovalManager solves claudy's trap for free when an operator is connected" is **false on the server** — there is no decision return-wiring for the core manager there at all (Validation Notes Correction 4). The guard is the only thing preventing a permanent `balanced` hang or a fail-open `high_speed` auto-approve on headless runs.

4. **`allowedPrompts` → scoped session grants via the existing `rememberDecision()` + risk-ceiling (reuse, don't invent).** `SubmitPlanForReview.allowedPrompts: {tool, action, domain?}[]` (browser-domain vocabulary: `{tool:'browser_dom', action:'submit', domain:'shop.com'}`, not claudy's `{tool:'Bash', prompt}`). On approval, seed each via `ApprovalGate.rememberDecision(tool, params, 'auto_approve', domain, riskScore)` (`:275-291`) so execution does not re-prompt; the existing `RISK_CEILING_MARGIN=25` (`:170`) guarantees an escalated variant still re-prompts. **Reuse**, exactly as `RepublicAgentEngine.handleExecApproval`'s `remember` path already does (`:697-706`). Grants are keyed by `buildMemoryKey` (tool+action+domain — already `browser_dom`/`browser__*`-aware, `:400-418`), so they are inherently domain+action scoped.

5. **Plan staleness — declared per step, re-validated by model discipline (advisory, not code-enforced).** Browser DOM is not a stable filesystem; page state changes between plan and execution. Each *mutating* step carries a lightweight `precondition` (URL substring / selector present) in the plan schema; the approved-plan echo instructs the model to re-verify it with read-only tools before that step and, on mismatch, stop and re-submit a revised plan. **There is no execution interceptor** in browserx that can hard-enforce "before step N" — the freeze is intentionally lifted post-approval — so this is best-effort by design (claudy, on a stable filesystem, doesn't need it at all). See Phase 4 for the honest enforcement boundary and the optional `verify_precondition` hardening.

6. **Editable plan reuses the *existing* approval round-trip — Phase 3 is mostly free.** The webfront approval card already exposes **`onRequestChange(text)`** → `sendApprovalDecision(id,'reject',false,text)` → `ExecApproval` op with `alternativeText` → `RepublicAgentEngine.handleExecApproval` already plumbs `alternativeText` into the `ApprovalResponse` (`EventProcessor.ts:733,777-796`). So a user-edited plan rides back through this **already-wired** channel with **zero new ops**: `SubmitPlanForReview` reads the edited plan off the resolved `ApprovalResponse` (`.reason`/`.modifications`), re-persists the artifact (`edited`), and labels the echo `"(edited by user)"`. Net-new work is only the *richer card UI* (render structured steps + an edit affordance) — not the transport. A distinct `type:'plan_review'` (widen the closed 5-union at `ApprovalManager.ts:12` + `ApprovalGate.mapToolToApprovalType()` `:423-430`) is an optional Phase-3 nicety for a dedicated card style; Phases 1–2 use the existing `'dangerous_action'` type and carry no UI-union dependency.

7. **Coordinator/teammate plan-approval maps to Track 04 (Track 06 ABANDONED 2026-05-14).** Claudy's leader-mailbox path maps to Track 04's `SubAgentRegistry` (`SubAgentRegistry.activeAgents`) + its management tools `sub_agent` / `send_message` / `cancel_sub_agent` / `list_sub_agents`, scoped per session. Do not design it here (see Track 06).

8. **Persistent read-only guidance via a PromptComposer fragment + `/plan` as a `PromptCommand` (the model-facing surface — what makes it work *across turns*, not just one tool call).**
   - **The cross-turn problem.** Plan Review spans many exploration turns before `SubmitPlanForReview`. A single `BeginPlan` tool-result instruction will not reliably hold over N turns. The **hard** correctness guarantee is the freeze (Decision 1: every non-read-only call is denied with `reason:'plan-review-freeze'`, so the model *cannot* mutate and *learns* from the denial); the freeze alone is sufficient for safety. The **soft** guidance (so the model doesn't waste turns attempting denied actions) is browserx's analog of claudy's recurring attachment: a new prompt fragment.
   - **Mechanism.** `PromptComposer.composeMainInstruction(agentType, context)` (`src/prompts/PromptComposer.ts:56`) assembles the system prompt from `.md` fragments **fresh every call** (via `PromptLoader`). Add `planReviewActive?: boolean` to `RuntimeContext` (`:24-43`), a new `fragments/plan_review.md`, and `if (context?.planReviewActive) sections.push(planReview)` in `composeMainInstruction`. Because it is re-composed each turn and gated on a `RuntimeContext` boolean, the instruction persists for the entire review and is **orthogonal to the reserved agent-`mode`** (it does not touch `MODES`/`composeMainInstruction`'s mode axis — PR 223/228 ownership is unaffected). `PromptLoader` learns `planReviewActive` from `registry.planReviewActive` where it builds `RuntimeContext`.
   - **`/plan` is a `PromptCommand`, not a gate toggle.** `Command = PromptCommand | LocalCommand` (`core/commands/types.ts:82`); `PromptCommand.getPromptForCommand(args,ctx)` (`:69`) returns text injected as the turn prompt. `/plan <task>` → returns *"Enter plan review for this task: call `BeginPlan`, explore read-only, then present a plan via `SubmitPlanForReview`. Task: <args>"*. The **model** then drives `BeginPlan`, which is what makes the freeze + fragment coherent (a silent gate flip without the model knowing would just produce confusing denials). Fully consistent with Track 03/13 (slash → funnel → prompt); **no new agent op**. Exit is by the review decision (approve/reject both `endPlanReview()`); `/plan` with no args while active is a no-op-with-hint.

### Data model (shared by all four phases — decide once)

The model **cannot author the plan via a tool**: `planning_tool` write commands are non-read-only and therefore frozen during review (Classification Audit). Claudy's incremental plan-*file* does not port (no project FS). Resolution: **the plan *is* the `SubmitPlanForReview` input** — the model explores read-only, composes the plan in reasoning, submits it whole. One structured schema serves the echo (P2), the editable card (P3), staleness + grants (P4):

```ts
// SubmitPlanForReview input
{
  summary: string;
  steps: Array<{
    description: string;
    mutating: boolean;                       // model's own classification, for the card + P4
    precondition?: { urlIncludes?: string; selectorPresent?: string };  // P4 staleness
  }>;
  allowedPrompts?: Array<{ tool: string; action: string; domain?: string }>;  // P4 grants
}
```

New `RolloutItem` variant (closed union at `src/storage/rollout/types.ts:162-169`; parallel `RolloutItemRecord.type` enum `:179`; type-guard pattern `:341`; zod schema in `src/storage/rollout/RolloutWriter.ts`; write/read in `RolloutRecorder.ts`):

```ts
| { type: 'plan_artifact'; payload: {
    planId: string; sessionId: string; turnId: string; createdAt: number;
    status: 'submitted' | 'approved' | 'rejected' | 'edited';
    plan: <the SubmitPlanForReview input>;
    editedBy?: 'user';
    prePlanSequence: number;   // rollout sequence of the turn before BeginPlan — Track 15 rewind anchor
  } }
```

Persisted via `RolloutRecorder.recordItems([...])` on submit (`submitted`) and on decision (`approved`/`rejected`/`edited`). Track 15's `rewindToMessage(sourceConversationId, prePlanSequence)` discards it on rejection. **Net-new, not a drop-in** — but every touch point is enumerated above.

### Phase 1 — Classification Audit (the hard gate): findings & checklist

> Completed 2026-05-15 by direct read of every registrar + the implementation of every tool that declares `isReadOnly:true`. This is the audit itself, not a request to perform one — an implementer applies the **Fixes** column, no further investigation required.

**Pass/fail criterion (apply to every `isReadOnly:true`).** A tool/input may be labelled read-only for the freeze **only if its implementation provably performs zero side effects**: no DOM mutation, no navigation / `chrome.tabs.create|update`, no network write (POST/PUT/DELETE/payment), no storage/cache/setting write or delete, no download, no clipboard write, and **no arbitrary injected script (`chrome.scripting.executeScript`) against an ambient, non-session-bound tab** (an injected "read" script is not *provably* a read). Anything failing this is **too-permissive (a freeze bypass — security)**; a genuine read wrongly frozen is **too-restrictive (usability)**.

**Fail-closed safety net (verified, so the audit scope is bounded).** `ToolRegistry.isReadOnly` returns `false` when: the tool is unregistered (`if (!entry) return false`), the tool has no `runtime` profile (`register()` merges `DEFAULT_TOOL_CONCURRENCY_PROFILE` whose `isReadOnly:()=>false`), or the predicate throws (`catch { return false }`) — all → **frozen**. Therefore the *only* freeze-bypass risk is an explicit `isReadOnly:true` declaration or an MCP `readOnlyHint:true`. The table below is the complete set of those.

## Risks

- **Freeze soundness depends entirely on Track 02 `isReadOnly` accuracy.** The audit is **complete** (see "Phase 1 — Classification Audit"); residual risks are the two it surfaced: (a) MCP `browser__*` read-only status is the MCP server's *self-declared, unverified* `readOnlyHint` — mitigated only by the mandatory built-in-server allowlist fix; (b) `data_extraction` runs arbitrary injected script against an ambient tab and must stay force-frozen until Track 27. - **`ApprovalGate` is not constructed on the server, and the core `ApprovalManager` has no server return-wiring.** Headless Plan Review is registration-gated OFF; if a future Track-20 policy enables it, the platform-uniform `ToolRegistry` freeze (Decision 1) is what holds there, and the policy must resolve the approval. Do not re-introduce the "ApprovalManager solves the trap for free" assumption.
- **`high_speed` + any path that reaches the core manager auto-APPROVES on the 10-min timer (fail-open).** The freeze must hard-deny so a frozen mutation never reaches the manager at all. - **Scoped grants must stay tightly bounded (tool+action+domain).** Reuse the risk-ceiling margin (`:170`) so an escalated action still re-prompts; never widen `buildMemoryKey` for plan grants.
- **Plan staleness** — browser DOM is not a stable filesystem; worse headless. Phase 4 precondition re-validation is mandatory for mutating steps.
- **`ToolRegistry.planReviewActive` is process-/registry-global.** Multi-session/multi-agent (Track 04) needs it scoped per session/agent — flag for the scoping design (a `Map<sessionId,boolean>` on the registry, keyed off `request.sessionId` in `execute()`).
