**Priority: P1** · **Effort: M** (core + ext/desktop) → **M–L** end-to-end (incl. server, scheduler, Plan Review) · **Status: READY TO IMPLEMENT — every decision resolved, runtime-traced, hand-off-ready**

## Problem

When an agent run goes wrong there is no user-facing way to rewind to a prior turn and continue. The capability is *scaffolded but unreachable and broken*: an `InitialHistory {mode:'forked'}` variant exists, but (a) the only reachable code path persists an empty session, (b) the one method that implements fork correctly has zero callers, (c) `AgentRegistry` has no way to request a forked session, and (d) there is no command, UI, protocol method, or API to invoke any of it. On Apple Pi Server this is a missed operational lever: a failed unattended scheduled job cannot resume from its last good checkpoint — it can only re-run from scratch.

> claudy's `thinkback`/`thinkback-play` is a "Year in Review" **marketing animation, not rewind** — disregard. The real feature is `/rewind`.

### Storage model (forces fork — confirmed)

`RolloutWriter.addItems` (`RolloutWriter.ts:49-69`): append-only, `sequence:currentSequence++` — **per-item, monotonic per rollout**; new fork rollout restarts at 0 → target sequences are source-relative. `RolloutStorageProvider` (`provider/RolloutStorageProvider.ts:25-49`) has only whole-rollout delete — **truncate impossible; fork is the only model**. Tauri does **no JS-side sort** (`TauriRolloutStorageProvider.ts:80-91`) → slice **must sort by `sequence` defensively**. `RolloutRecorder.loadAllItems` maps to `{type,payload}` and **drops `sequence`** → slice & turn-list must call `provider.getItemsByRolloutId` directly. `compacted` (`CompactedItem={message}`) **never written today**; `summarize_up_to` is its first (transport-only) producer. `{type:'turn_completion'}` declared but **zero production writers** (`RolloutRecorder.recordTurnCompletion:258-275` dead) — **never use it as a checkpoint marker** (D10). Within a turn, `function_call` and its `function_call_output` are persisted in **one atomic `recordConversationItemsDual` batch** (`TaskRunner.ts:730-732`) before the next user item, and an aborted turn persists **nothing** (`TaskRunner.ts:357-366,414-429`) — so a user-boundary slice never bisects a pair (proof for D2), but an *assistant*-boundary slice can (D10/D11).

## Resolved Decisions (every prior open item + every runtime defect is closed)

- **D1 — Fork, not truncate; fix the broken fork first** (Phase 0a). *Divergence:* storage-layer fork vs claudy's in-memory slice.
- **D2 — `targetSequence` = a user `response_item.sequence`** (`payload.type==='message' && payload.role==='user'`; text at `payload.content[0].text`). Slice = `sequence <= targetSequence`, sorted ascending, source `session_meta` removed. Provably never bisects a tool pair (atomic-batch invariant).
- **D3 — Trigger ≠ claudy's command.** No `skip` type, no aliases → two webfront commands `rewind`+`checkpoint` sharing one `action`. Server gets an RPC.
- **D4 — Svelte selector (ext+desktop); server has none.** Clone `ChatHistoryPopup`→`PopupCard`→`ChatHistoryList`. Options: `conversation` / `summarize_up_to` / `nevermind`.
- **D5 — `summarize_up_to` compacts the ENTIRE `≤targetSequence` slice into ONE `compacted` transport item.** A rewind discards the tail, so there is no kept tail. Forked rollout seeded with exactly `[{type:'compacted',payload:{message:summary}}]`. *Intentional divergence from claudy* (prompt-cache-stable; consistent with reconstruct→one-system-message).
- **D6 — Code/artifact axis = desktop-only, P3.** Rewind restores the *conversation*, never side effects; server/UI/scheduler must state exec'd commands / file writes / connector sends are **not** undone.
- **D7 — No pre-abort needed.** `removeSession` aborts the in-flight turn and cleans up before returning (verified). Rewind mirrors `session.resume`'s `removeSession`→`createSession` sequence.
- **D8 — Continuation semantics (deliberate per-trigger divergence):** ext/desktop `conversation`: forked session waits for input; set `Main.svelte inputText` = rewound-to user turn's full text **after** the reset/restore (else clobbered, `Main.svelte:841,1189`); the handler returns this text. ext/desktop `summarize_up_to`: leave input untouched. **server/scheduler: re-submit programmatically** (headless win).
- **D9 — modelClient via DI (mirrors `loadRolloutHistory`).** `computeRewindSlice` stays pure. Summarization is optional `SessionServiceDeps.summarizeForRewind?:(items:ResponseItem[])=>Promise<string|undefined>`, guarded, wired per-platform from each platform's per-agent `ModelClientFactory`, invoked **before** `removeSession`.
- **D10 — Scheduler checkpoint = the greatest USER `response_item.sequence` that is `≤ max(assistant response_item.sequence)`** — NOT the assistant sequence directly. (`turn_completion` is dead code — never used.) Backing up to the user-turn boundary keeps only fully-batched turns and is the only dangle-free boundary (atomic-batch proof). If no user item exists (job failed before first user/assistant pair), no checkpoint → fall through to a normal bounded retry from the original `input` (retry-from-zero); the fork win materializes only for jobs with ≥1 completed turn. Hook lives **inside `JobExecutor.failExecution`, before `cleanupSession` (`:219`)** — the single chokepoint covering launcher-thrown and event-driven failures. Retries bounded by a new `MAX_REWIND_RETRIES` constant; failed-attempt side effects are **not** undone — log loudly.
- **D11 — `computeRewindSlice`/`buildSummarizedFork` MUST run a `call_id` pairing trim** after slice + defensive sort, so *no* `targetSequence` (selector value, Plan-Review `BeginPlan` sequence, server/scheduler checkpoint) can yield a malformed first request. Build `completed = {function_call_output.call_id in kept}`; then **(a)** drop any standalone `function_call` whose `call_id ∉ completed`; **(b)** for any assistant `message` with `tool_calls`, strip each `tool_calls[]` whose `id ∉ completed` (drop the message if it then has no content); **(c)** drop any orphan `function_call_output` whose `call_id` has no call in `kept`. Pairing key: `call_id` (`protocol/types.ts:243,247`; unified `message.tool_calls[].id` `:213`). **Do NOT rely on the request-layer `processMissingCalls` (`TurnManager.ts:557-589`)** — it ignores the unified `message.tool_calls` shape, which is exactly the D10 dangling case. D2 (user boundary) is already dangle-free by construction; D11 is the belt-and-suspenders that makes the slice fn correct for *any* caller and any input shape.
- **D12 — Fork matches resume: `turn_context` is dropped (model/effort/approval not restored) in v1.** Restoring it *only* on fork would make fork inconsistent with resume (both use the same `reconstructHistoryFromRollout`), violating system consistency. So v1 fork = resume parity: the forked session runs with current global config (the approval-mode reset is toward the *safer* `on-request` default; the only real wart is a possible silent model change, surfaced in Risks). **Optional follow-up (out of Track 15 v1 scope, noted for the owner):** restore the last pre-slice `turn_context` (approvalPolicy/model/effort) at the shared `recordInitialHistory` seam — this would upgrade *both* resume and fork together, preserving consistency. Not done in v1 to keep the track scoped and consistent.
- **D13 — Source rollout MUST be flushed before any read.** The static read path bypasses the live writer (`writeQueue`), so the selector/slice can miss the newest in-flight turns. Every rewind entry point, before calling `listUserTurns`/`computeRewindSlice` on a live source conversation, must `await registry.getSession(sourceConvId)?.agent?.getSession()?.flushRollout?.()` (guarded — no-op if the source session is not live). This requires a tiny net-new public `Session.flushRollout()` (Phase 0c).
