# Implementation Plan: Enrich Scheduler Page

## Summary

Redesign the scheduler page from a single schedule-a-job form into a full-featured three-module page (Active Jobs, Job History, New Job) with fuzzy search, sort/filter, repeat/recurrence support, and responsive wide/narrow layout. The implementation extends existing Svelte components, Fuse.js patterns, IndexedDB storage, and the `isWideMode` layout store.

**Language/Version**: TypeScript 5.x, Svelte 4
**Primary Dependencies**: Svelte 4 (UI), Fuse.js 7.1.0 (fuzzy search), Tailwind CSS (styling), chrome.runtime messaging, svelte-spa-router
**Storage**: IndexedDB (`scheduler_jobs` store) via SchedulerStorage, chrome.storage.local (scheduler state)
**Testing**: Vitest with jsdom environment, Chrome API mocks
**Target Platform**: Chrome Extension (Manifest V3) + Tauri desktop app
**Project Type**: Single project (extension/desktop hybrid)
**Performance Goals**: Search debounce 150ms, sort/filter instant (client-side), recurrence calculation < 10ms
**Constraints**: Chrome alarms API minimum 1-minute granularity, sidepanel UI constrained width (~400px narrow, 1500px+ wide)
**Scale/Scope**: Typical user has 10-100 historical jobs, 0-10 active jobs

## Key Design Decisions

### 1. Page Layout Architecture

The scheduler page becomes a responsive container for three modules:
- **Wide mode** (`$isWideMode`): 3-column grid - Active Jobs (left), New Job (center), Job History (right)
- **Narrow mode**: Vertical stack with collapsible section headers - New Job (top, expanded), Active Jobs (middle), Job History (bottom)

Uses the existing `isWideMode` readable store from `layoutStore.ts` (breakpoint: 1500px).

### 2. Job History Search Strategy

Reuse the Fuse.js pattern from `SettingsSearch.svelte`:
- Build Fuse index on `input` field of archived jobs (weight: 2) and `status` (weight: 1)
- 150ms debounced search
- Client-side filtering on the fetched page of results
- Pagination interacts with search: when search is active, client-side filter the full loaded set; "Load More" fetches the next page and appends to the Fuse index

### 3. Sort Implementation

Sort is purely client-side on the already-fetched archived jobs array:
- **Newest first** (default): Sort by `completedAt` descending
- **Oldest first**: Sort by `completedAt` ascending
- No additional API calls needed - just `Array.sort()` on the cached data

### 4. Status Filter Implementation

Multi-select dropdown/chip filter:
- Options: `completed`, `failed`, `cancelled`
- All selected by default
- Applied client-side after search, before rendering
- Visual: Inline pill/chip buttons that toggle on/off

### 5. Recurrence Data Model

Add to `SchedulerJobRecord`:
```typescript
recurrence?: RecurrenceRule | null;
```

New `RecurrenceRule` interface:
```typescript
interface RecurrenceRule {
  mode: 'daily' | 'weekly' | 'monthly' | 'custom';
  interval?: number;        // For custom: every N units
  intervalUnit?: 'minutes' | 'hours' | 'days' | 'weeks'; // For custom
  endCondition: 'never' | 'after' | 'until';
  endAfterCount?: number;   // For 'after': total occurrences
  endUntilDate?: number;    // For 'until': Unix timestamp
  completedCount?: number;  // Track completed occurrences
  parentJobId?: string;     // Link to original job in chain
}
```

### 6. Recurrence Execution Flow

When `Scheduler.completeJob()` or `Scheduler.failJob()` is called for a job with a recurrence rule:
1. Check if recurrence should continue (not cancelled, end condition not met)
2. Calculate next scheduled time using `recurrence.ts` utilities
3. Create a new `SchedulerJobRecord` with the same input, next scheduled time, and updated recurrence (incremented `completedCount`)
4. Set alarm for the new job

### 7. Backwards Compatibility

- `recurrence` field is optional on `SchedulerJobRecord` - existing jobs without it are treated as one-time
- No IndexedDB migration needed - new field is simply added to records going forward
- Existing `ArchivedJobsView` and `SchedulerPopup` remain functional without changes

## Data Model

### RecurrenceRule (new)

| Field | Type | Description |
|-------|------|-------------|
| mode | `'daily' \| 'weekly' \| 'monthly' \| 'custom'` | Repeat frequency type |
| interval | `number?` | Custom interval value (e.g., every 2) |
| intervalUnit | `'minutes' \| 'hours' \| 'days' \| 'weeks'?` | Custom interval unit |
| endCondition | `'never' \| 'after' \| 'until'` | When the recurrence stops |
| endAfterCount | `number?` | Total occurrences (for 'after') |
| endUntilDate | `number?` | End date timestamp (for 'until') |
| completedCount | `number?` | How many occurrences have completed |
| parentJobId | `string?` | ID of the original job that started the chain |
