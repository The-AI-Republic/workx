**Input**: Feature specification from `/specs/037-scheduler-calendar-view/spec.md`

## Summary

Add a calendar view page (`/scheduler/calendar`) to the scheduler using `@event-calendar/core` v5.x. The project's `package.json` and lockfile target Svelte 5 (`^5.53.7`), and Svelte 5 has backwards compatibility with Svelte 4 component syntax (the existing codebase uses Svelte 4 patterns but runs on Svelte 5). The scheduler page gets a "Calendar View" button that navigates to the calendar, where users can visualize, create, and edit scheduled jobs on day/week/month views. The implementation adds new npm dependencies, a new route, a new page component, and a calendar-job mapping layer.

## Key Design Decisions

### 1. Package Version Selection

The project's `package.json` declares `svelte: "^5.53.7"` and the lockfile resolves to 5.53.7. The installed `node_modules/svelte` may still be 4.2.20 on some machines (stale install), but a fresh `npm install` will install Svelte 5. Svelte 5 provides full backwards compatibility for Svelte 4 component syntax, so existing `.svelte` files work unchanged.

Using v5.x (latest, Svelte 5 native):

| Package | Version | Purpose |
|---------|---------|---------|
| `@event-calendar/core` | ^5.4.1 | Calendar engine |
| `@event-calendar/day-grid` | ^5.4.1 | Month view |
| `@event-calendar/time-grid` | ^5.4.1 | Day/week views |
| `@event-calendar/interaction` | ^5.4.1 | Click, select, drag-and-drop |
| `@event-calendar/list` | ^5.4.1 | List view (optional fallback) |

### 2. Routing Strategy

`svelte-spa-router` supports flat routes. Add `/scheduler/calendar` as a new route in `App.svelte` pointing to `SchedulerCalendar.svelte`. No nested router needed.

```typescript
const routes = {
  '/': Chat,
  '/settings': Settings,
  '/scheduler': Scheduler,
  '/scheduler/calendar': SchedulerCalendar,  // NEW
  '/skills': Skills,
  '*': Chat,
};
```

The `/scheduler` route must be listed AFTER `/scheduler/calendar` to prevent prefix-matching issues, OR use `svelte-spa-router`'s exact matching (which is the default behavior for hash-based routing).

### 3. Job-to-Event Mapping

Pure utility function in `calendarUtils.ts`:

```typescript
function jobToCalendarEvent(job: SchedulerJobRecord): CalendarEvent | null {
  if (!job.scheduledTime) return null; // Skip drafts
  return {
    id: job.id,
    start: new Date(job.scheduledTime),
    end: new Date(job.scheduledTime + 30 * 60 * 1000), // Default 30min duration
    title: job.input.slice(0, 50),
    backgroundColor: statusToColor(job.status),
    extendedProps: { job },
  };
}
```

Status-to-color mapping:
| Status | Color (Modern) | Color (Terminal) |
|--------|---------------|-----------------|
| scheduled | `#22c55e` (green) | `#00ff00` |
| running | `#3b82f6` (blue) | `#00ffff` |
| waiting/queued | `#60a5fa` (light blue) | `#00cccc` |
| missed | `#eab308` (yellow) | `#ffff00` |
| failed | `#ef4444` (red) | `#ff4444` |
| completed | `#6b7280` (gray) | `#666666` |
| cancelled | `#9ca3af` (light gray) | `#444444` |

### 4. Calendar Theme Integration

The `@event-calendar/core` supports CSS variable customization. Create theme-specific CSS overrides:

- **Modern theme**: Use existing `--chat-*` CSS variables for backgrounds, borders, text
- **Terminal theme**: Override with terminal green palette, monospace fonts, dark backgrounds

Apply via a wrapping `<div>` with theme-conditional classes that set CSS custom properties consumed by the calendar.

### 5. Data Fetching Strategy

The calendar page fetches ALL jobs (not just active) to show historical events too:
- Fetch active jobs (running, scheduled, queued, missed) via existing messages
- Fetch archived jobs (completed, failed, cancelled) via `SCHEDULER_GET_ARCHIVED_JOBS` with a date range filter
- New storage method `getAllJobsInRange(startTime, endTime)` returns all jobs whose `scheduledTime` falls within the visible calendar range
- Re-fetch when calendar date range changes (view change, navigation)

### 6. Reschedule Implementation

New `RESCHEDULE_JOB` message type that updates `scheduledTime` on an existing job:
- Only allowed for `scheduled`, `missed`, `draft` statuses
- Updates the record in IndexedDB
- Clears old alarm, creates new alarm for updated time
- Emits `JOB_STATUS_CHANGED` event to update all listeners

### 7. Interaction with Feature 036

This feature is designed to work both before and after feature 036 (enrich-scheduler-page):
- If 036 is implemented first, the "Calendar View" button goes into the redesigned scheduler page
- If 037 is implemented first, the button goes into the current scheduler page/popup
- The calendar page itself is independent - it only reads job data via messages

### 8. Default View by Screen Size

Use the existing `isWideMode` store:
- Wide screen (>= 1500px): Default to `timeGridWeek` view
- Narrow screen: Default to `timeGridDay` view
- User can always switch views manually via the view switcher
